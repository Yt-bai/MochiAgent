# ✈️从LLM进化到Agent

## 🏞️ 能干实事的好员工

### 🏙️ Context Engineering（上下文工程）—— “动态组装信息”

每次 Agent 思考时，需要把所有相关信息组装成一个“完整的上下文”。

<img width="890" height="620" alt="image" src="https://github.com/user-attachments/assets/b77e71a0-eae6-4cc4-b8e8-d7ef5adb9c02" />

**context** vs **memory** vs **context engineering**

**“context（上下文）”是结果，memory 是其中一种输入来源，context engineering 是把这些输入挑选并拼装成 context 的方法/系统。**

## 1️⃣ Context（上下文）是什么？

在 LLM / Agent 里，**context = 这一次模型推理时“实际看到的输入”**。

通常包括：

- system prompt（规则 / 身份）
- developer prompt（产品策略 / 风控）
- 当前用户消息
- 对话历史（短期）
- 工具返回结果
- RAG 检索出来的资料
- memory（长期偏好 / 画像 / 关键事实）

📌 **关键：context 是每一次调用模型时的“最终拼装包”。**

---

## 2️⃣ Memory 是什么？（context 的一个来源）

**Memory = 可跨回合 / 跨会话保存的用户或任务相关信息，用于后续对话更好地理解与个性化。**

常见两类：

### 短期记忆（Short-term Memory）

- 本次会话内的对话历史  
- 很多系统也把它算 memory，但工程上经常叫 *conversation history*

### 长期记忆（Long-term Memory）

- 跨会话保存
- 用户偏好
- 身份信息
- 历史行为摘要等

📌 **关键：memory 本身不是上下文，它是“材料”。只有当它被选中并注入 prompt 时，才进入本次 context。**

---

## 3️⃣ Context Engineering（上下文工程）是什么？

**Context engineering = 决定“哪些信息进 context、以什么格式进、进多少、放在哪”** 的工程方法 / 系统（动态组装）。

它要解决的问题是：

- context window 有限（装不下全部）
- 装太多会噪声大、变慢、变贵
- 装错顺序 / 格式会导致模型忽略关键信息
- 需要按任务动态选择  
  - 查订单 → 要订单信息  
  - 写文案 → 要品牌规范  

📌 **关键：上下文工程的产物是“最终 context”。**

---

## 4️⃣ 三者关系（一句话 + 类比）

### 一句话总结

- 🌈**Memory**：可复用的信息库存  
- 🌈**Context engineering**：拣货 + 打包策略（筛选、压缩、排序、模板化）  
- 🌈**Context**：打包后的包裹（这次实际喂给 LLM 的内容）

---

### 类比（更直观）

- **Memory** 像“用户档案 / 历史记录”
- **Context engineering** 像“秘书整理材料”
- **Context** 像“递到你桌上的文件夹”
