### ⛵️ Tools —— 给它“双手”

如果用户说“帮我查一下我的订单”，这需要真正去调用订单系统的 API，而不只是查知识库。

**Tools（工具）** —— 简单说，Tool 就像给 LLM 配了一套工具箱： 

查订单、退款、修改地址、发优惠券……每个工具都是一个可以执行的功能。

---

```text
用户: "帮我查一下订单 12345 的物流"

LLM 判断: 需要调用 "查询物流" 工具

系统调用: get_logistics(order_id="12345")

返回结果: 商品已发货，预计明天送达

LLM 整理回复: "您的订单已发货，预计明天送达，快递单号是 SF1234567890。"

```

🍁🍁🍁**tool 调用的关键技术：Function Calling**🍁🍁🍁

这个术语听起来很技术，但本质很简单：LLM 现在能“告诉你它想调用哪个工具”。

#### 🌊Function calling的发起者

模型（LLM）发起。

准确说：在你请求模型之后，模型在响应里返回**tool_calls函数**，表示“我建议调用某个工具”。

>> 这一步是“模型发起调用意图”，不是模型执行。

#### 🌊Function calling的接收者

你的应用/后端（Orchestrator / Tool runner）接收。

也就是服务端中间层读取 message.tool_calls：

• 解析出函数名 get_logistics

• 解析出参数 {"order_id":"12345"}

• 决定是否允许调用（权限/风控）

• 然后去执行真实函数/API

#### 🌊Function calling的执行者

还是你的应用/后端在执行（或它调用的下游服务在执行）。
 
🚦不用 MCP：你自己的 service layer 直接执行/调 API🚦

<img width="685" height="231" alt="image" src="https://github.com/user-attachments/assets/43b91190-4971-46a9-a057-065543cbf862" />

🚦用 MCP：你的 tool runner 调 MCP server，让 MCP server 去执行/调API（执行仍然不在模型里）🚦

<img width="804" height="360" alt="image" src="https://github.com/user-attachments/assets/26d907eb-54c0-4cc5-9704-75e247ca7c48" />

MCP让“工具如何被提供与调用”标准化，不是必须，但在多工具生态里很有价值

#### 🌊工具调用结果返回

后端返回role="tool"内含使用的工具和工具调查到的答案结果，返回给模型（LLM）。

--------------------------------------------------------------------------------------------------
### 这里不得不再提到Prompt和Tools schema，因为这两个是配合Function calling工作的前置工作。

❄️❄️System Prompt：告诉模型“什么时候该用工具、什么时候不该用、选工具的偏好、缺参怎么办”等（决策规则）

❄️❄️Tools schema：给LLM模型看的“工具说明书”，告诉模型“有哪些工具、每个工具需要什么参数、参数类型约束”（可选空间 + 约束）

❄️❄️Function calling 机制：提供一个“结构化输出通道”（tool_calls），tool_calls 是专门给程序解析的字段，让模型把决策结果告诉程序，让执行程序能看懂。

**🐾🐾🐾工具调用流程：用户问 → 模型返回 tool_calls（想调用什么） → 后端/中间服务层/MCP执行工具（真干活） → 后端把结果用role="tool"消息回传 → 模型整理成最终回复**

## 这就是 LLM 和 Agent 的本质区别：

- **LLM** – 只能“说”，像一个顾问。你问它问题，它给你建议，但不会帮你执行。  
- **Agent** – 能“说”也能“做”，像一个员工。它不仅告诉你怎么办，还能真的帮你办了。

---

| 维度 | LLM | Agent |
|------|------|--------|
| 能力 | 只能“说” | 能“说”也能“做” |
| 类比 | 顾问 | 员工 |
| 工具 | 无 | 有（API、数据库等） |
| 自主性 | 被动回答 | 主动执行任务 |
